## Run conditions

- **Date**: 2026-05-10
- **Machine**: Intel Core i5-4570 @ 3.20GHz (Haswell, 4c/4t, no SMT), Windows 10 host
- **Runtime env**: WSL2 Ubuntu 24.04 (LTS), kernel 6.6.114.1-1
- **CSV location**: `/mnt/c/Users/Rick7/code/arche/design_analysis/benchmarks/etl/data/data_100m.csv`
  - accessed through the WSL2 9p protocol filesystem bridge from the Windows host
- **Dataset**: 100,000,000 rows, ~35 bytes/row, 3.4 GB; generated by `data/generate_data.py`
- **Iterations**: 3 per engine; reported numbers are `min`
- **Runner**: `compare_scale.py --task all --engines arche,pandas,polars,duckdb,datafusion --iterations 3`
- **Arche commit**: see `git log -1` at the time this file lands

| Engine | Version | Threading model |
|---|---|---|

## Theorized reasons for the slowdowns

### 1. `/mnt/c` 9p protocol is the dominant cost for I/O-heavy workloads

WSL2 reads from `/mnt/c` go through a 9p protocol bridge to the Windows
filesystem, which adds significant per-syscall latency vs native ext4 reads.
Sequential reads see roughly 5-10x worse throughput than native ext4 on the
same physical disk. Arche's `csv_mmap_open` + sequential byte-walk pattern
generates many small page faults, each of which takes the slow path through 9p.

This hits Arche hardest because:
- Arche is single-threaded; one thread waiting on a 9p syscall blocks the whole
  parse loop
- Arche's parse path does many tiny reads/parses per row (find comma, parse
  float, find next comma, parse int, advance) instead of a bulk SIMD scan

### 2. Multi-threading hides 9p latency for Polars / DuckDB / DataFusion

Polars, DuckDB, and DataFusion all default to multi-threaded CSV scans. With 4
physical cores and the parse-bound 9p I/O, threads overlap their stalled reads
and stay closer to ext4-native throughput. Their measured slowdown vs the
README's Linux numbers is therefore much smaller (or even nil - DuckDB at 18s
across all four tasks is close to its native-Linux ceiling on this CPU).

Pandas's CSV scan is single-threaded, so it's somewhere in the middle of the
slowdown spectrum.

### 3. The i5-4570 (Haswell, 2013) is a modest CPU

The README doesn't specify the CPU it used, but anything in the last 5 years
will outperform Haswell at single-threaded float parsing by ~2x. This shows up
as a baseline tax across all engines - not just Arche - but it stacks on top of
the 9p penalty for compute that doesn't parallelize well.

### 4. Arche has no parallelism and no SIMD on the parse path

Arche's column ops (the `Transaction.revenue = Transaction.price * Transaction.quantity`
step) are vectorized to SSE/AVX in the LLVM backend. But the parse loop that
fills the columns is a scalar `for` loop calling `csv_mmap_parse_float` /
`csv_mmap_parse_int` one row at a time. With 100M rows that's 100M function
calls in a tight loop with no opportunity for SIMD. On a fast filesystem this
is still bounded by parser CPU; on the slow 9p mount it's bounded by page-fault
latency.

### 5. Task 2 specifics

Task 2 was the README's biggest arche win (10.1x vs pandas). Here it's 1.59x.
Two reasons:
- The `usecols=['quantity']` change to the pandas script (committed earlier in
  this branch) removed an unfair advantage Arche had - pandas now skips parsing
  the 3 columns it doesn't need. That accounts for a chunk of the gap.
- The 9p penalty still applies, but it applies to pandas too, so the relative
  ratio compresses.

So this is actually a *fairer* Task 2 number than the README has, even if it
looks worse for arche.

## Same-machine vs cross-machine takeaway

The README's headline 4-10x arche-over-pandas claims **do not survive**
same-machine measurement with the fairness fixes applied. The order of magnitude
of the gap is correct (compiled is faster than interpreted), but on a level
playing field:

- DuckDB beats arche by ~2-3x on this hardware/OS
- Polars and DataFusion sit between arche and DuckDB
- Pandas+usecols is within 1-2x of arche on three of the four tasks
- Arche's lead is real but smaller than the README suggests

The right next step for honest headline numbers is a **native Linux** rerun on
this same hardware (drop WSL entirely; either dual-boot or move to a Linux
machine). Until then, this file is the most truthful numbers we have.

## What this run does NOT yet cover

The codebase now supports several improvements that this run predates - they
were added during this work but the run itself used the older configuration.
A re-run with the full setup would also include:

- **Hand-written C baseline** (`c_baseline/`) - should set a near-floor for what
  any compiled mmap-based parser can achieve on this dataset
- **Task 5 multi-step pipeline** (`task_5_pipeline`) - filter q>0 AND price>10,
  then sum - exercises the engines' real pipeline planning
- **10 iterations + min/median/max** - quantifies variance instead of trusting
  single min
- **`--cold-cache` flag** - drops the OS page cache between iterations to
  measure cold I/O cost, not just hot-cache parse+compute (requires NOPASSWD
  sudo setup on Linux/WSL)

A rerun command that covers all of the above:

```
python3 design_analysis/benchmarks/etl/compare_scale.py \
    --task all \
    --engines arche,c,pandas,polars,duckdb,datafusion \
    --iterations 10 \
    --csv design_analysis/benchmarks/etl/data/data_100m.csv
```

Add `--cold-cache` once the NOPASSWD line is in `/etc/sudoers.d/`.

## Active disclaimers

This run still doesn't address the following items from
`design_analysis/README.md`:

- **No multi-core parity**: Polars/DuckDB/DataFusion are multi-threaded;
  Pandas/Arche are not. The "compiled engines vs pandas" gap conflates
  compiled-code wins with parallelism wins. A fair single-thread variant would
  set the relevant engines to 1 thread.
- **No dirty data**: `data/generate_data_dirty.py` exists for this; the dataset
  used here is the clean generator's output, which doesn't stress quoted-field
  parsing or null handling.
- **No joins**: Task 5 covers filter→aggregate, not joins. A small dimension
  join would round out pipeline coverage.

## Codegen bug surfaced

Implementing the vectorized form of Task 5 in arche
(`revenue = revenue * (quantity > 0) * (price > 10.0)`) triggered a codegen bug
where the int comparison mask isn't lifted to `<4 x i32>` before being fed
into a vectorized `sitofp` for the float multiply.
(NOTE: now fixed; the writeup doc was removed and the repro relocated to
`tests/unit/language/types/int_mask_times_float_column.arche` as a passing test.)
Task 5 in arche fell back to a scalar `if`-filter loop as a workaround at the time.
