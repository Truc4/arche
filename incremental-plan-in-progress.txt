  The short answer

  Keep it in this repo, but not in stdlib/ ‚ÄĒ put it in a sibling top-level
  folder for first-party-but-optional devices. And call the library gfx.

  Why not a separate repo (yet)

  The decisive fact: arche has no package manager or external module resolution.
  Devices are imported by bare name (#import { gfx }) and resolve from the
  installed stdlib tree; everything else is imported by relative path. So a
  separate repo today could only be consumed by vendoring/submodule +
  path-import ‚ÄĒ which throws away the by-name device ergonomics and the
  .ds.arche datasheet model that make a device a device (docs/devices.md:10-33).
  You'd also be chasing a moving target: the language is alpha, and the FFI
  visibility bands + device model are still settling (docs/devices.md:90-111).
  An out-of-repo lib would break on every churn and you'd maintain both sides.
  In-repo, it rides the same build and the same refactors.

  A separate repo becomes the right call later ‚ÄĒ once there's a package manager
  and the FFI/device surface has stabilized.

  Why not stdlib/ either

  You're right to keep it out. stdlib is the "blessed," always-installed surface
  (make install lays down $PREFIX/lib/arche/stdlib, README:160). Graphics is
  heavy C-FFI (SDL/wgpu, or a framebuffer via the mmap path already noted in
  tests/unit/compiler/syscall/mmap_anon.arche) and orthogonal to arche's core
  DOD/ECS audience. Bundling it into stdlib commits you to installing and
  maintaining graphics with every build.

  The middle tier ‚ÄĒ in-repo, outside stdlib ‚ÄĒ is exactly the gap. It's a
  first-party device that ships in the tree (so it stays in sync and serves as a
  flagship device example: a Sprite/RenderTarget archetype + a present sys,
  with the driver owning the pool), but it isn't part of the installed blessed
  surface.

  On the name

  stdlib modules are short, lowercase, functional ‚ÄĒ fmt net io str term ‚ÄĒ never
  thematic (the language name is the only Greek flourish; modules stay plain).
  gfx fits that register exactly and reads well namespaced at call sites:
  gfx.clear(), gfx.present(), gfx.draw(...).

  The one thing genuinely yours to decide: the container directory name (since
  "not stdlib, but called what?" is really two names ‚ÄĒ the folder and the lib)
  and the lib name itself. Let me pin those down.

‚óŹ Fair pushback on both. Let me answer straight.

  Why contrib/

  It's not a strong claim ‚ÄĒ it's the closest-fitting convention, and there's a
  real connotation caveat you should weigh.

  The precedent: contrib/ is the established name for "first-party code that 
  ships in the source tree, sits outside the blessed core, and is opt-in."
  PostgreSQL is the textbook case ‚ÄĒ contrib/ holds extensions (hstore,
  pg_stat_statements) written by core devs, shipped in the repo, not compiled
  into the server unless you ask. Git has one too. That's exactly your
  situation: you wrote it, it rides in the repo, make install ignores it, a user
  opts in by vendoring it.

  Why not the others (from the subagent's repo scan): devices/ is redundant ‚ÄĒ
  everything in arche is a device, so the category name shouldn't be the unit
  type. lib/ collides with the install tree $PREFIX/lib/arche. extras/ is fine
  but a vaguer signal.

  The caveat: "contrib" literally reads as contributions ‚ÄĒ
  third-party/community. Yours is first-party-by-you. If that bugs you, extras/
  is the clean neutral runner-up. So this one's partly taste; I'll let you pick
  rather than insist.

  What the first pass actually renders

  I was vague in the plan ‚ÄĒ here's the honest scope, and it's forced by the 
  constraints, not a preference:

  First pass = 2D, CPU software rendering. No GPU. No 3D.

  Why it can't be otherwise yet:
  - No GPU / no 3D means no SDL, OpenGL, Vulkan, or wgpu ‚ÄĒ all of those are
  C/system libraries that need a link story. A vendored contrib/gfx/ folder
  copied into someone's project has no way to link external libs (the Makefile
  only links runtime/, and that's not installed for contrib). So anything
  GPU-backed is blocked until there's a build/link story ‚ÄĒ out of scope for a
  first cut.
  - That leaves a CPU framebuffer: a columnar pixel buffer (fits arche's
  whole-column DOD model perfectly), with clear / set_pixel / fill_rect /
  blit-type ops written as systems over the pixel columns. Pure arche +
  syscalls, zero external deps, so a copied folder just works.

  The one genuinely open question is where the pixels go ‚ÄĒ "present" needs a
  target, and the choices have real trade-offs (testability vs.
  actually-on-screen). That's your call.

‚óŹ Great instinct to slow down ‚ÄĒ and the survey turned up the fact that decides
  this. Let me lay out the honest layering, because "add openwindow to os" runs
  into a category problem and a language gap.

  "Open a window" isn't an OS thing

  On Linux there is no syscall to open a window. A window is one of two things:

  - A display-server client ‚ÄĒ X11 or Wayland. That's a protocol spoken over a
  unix-domain socket, then either you hand-roll the wire protocol or you link
  Xlib/libwayland. (And net today is AF_INET-only ‚ÄĒ runtime/net.c:27 ‚ÄĒ so even
  the socket is a gap.)
  - The framebuffer / DRM ‚ÄĒ /dev/fb0 (or DRM dumb buffers). This one is
  genuinely OS/syscall-level: open + ioctl for geometry + mmap the pixels.
  Full-screen, no window manager.

  So os.open_window() would mislabel a display-server client as an OS primitive.
  What genuinely belongs in the OS layer is the framebuffer: open /dev/fb0,
  ioctl its resolution, mmap it writable. That's the real "OS stuff," and it's
  the shortest path to "draw an image to a surface."

  The gap that actually blocks you (and forces a C shim)

  Here's the load-bearing finding: arche has no pointers, and there is no way to
  turn an mmap'd address into a writable char[] from arche code. os.syscall(9, 
  ‚Ä¶) already can mmap an fd ‚ÄĒ but it hands back a bare i64 address you can't
  index or write through. The only working pattern in the repo is io.file_map
  (runtime/io.c:33-54): a C shim does the mmap and returns a fat pointer as
  char[], which arche can then index and write byte-by-byte.

  Two consequences:

  1. Framebuffer access cannot be pure-arche. It needs a small C function in
  runtime/ (alongside io.c/net.c/term.c) that opens /dev/fb0, ioctls the
  geometry, mmaps it MAP_SHARED|PROT_WRITE, and returns (pixels: char[], w, h).
  runtime/ is the installed, linkable C layer ‚ÄĒ so this piece lands in stdlib + 
  runtime, not in a vendored extras/ device.
  2. That cleanly splits the two layers you were circling:
    - Surface acquisition (OS/C, installed): the framebuffer shim ‚ÄĒ call it an
  fb device (or fold into os). Returns a writable char[] + dimensions.
    - Drawing (pure arche, vendorable): extras/gfx ‚ÄĒ clear / set_pixel /
  fill_rect / blit / draw-an-image, all whole-column systems over a char[]
  surface. It doesn't care where the surface came from ‚ÄĒ framebuffer now, an
  image-file buffer, or a window later. This is the part the vendor-by-copy
  extras/ model fits.

  So your instinct is right, just relabeled: the OS layer gets a framebuffer
  primitive (not a window), and "draw the image to that" is the gfx layer
  drawing into the char[] the framebuffer hands it.

  The one caveat that gates feasibility

  /dev/fb0 is usually not writable under a running X/Wayland desktop ‚ÄĒ the
  compositor owns the display. It works on a bare TTY (Ctrl+Alt+F2), in a VM, or
  some efifb setups. So a framebuffer first pass produces real pixels, but
  you'd see them on a console, not inside your desktop session. That changes
  which target is actually satisfying ‚ÄĒ so it's the thing to pin down.

  The two asks collide at "the contract"

  world. You pick which .arche file gets merged, and ordinary name resolution
  catches a missing symbol. No contract needed ‚ÄĒ because the compiler sees both 
  sides' source at once.

  source while building the app. So it needs something describing the seam ‚ÄĒ a
  fixed symbol set + type layout. That "something" is a contract. It's the
  header you said you hate.

  So you can't literally have both "no contract" and "separate compilation." But
  you can get the thing you actually want, via one principle:

  ‚Ėé Generate the contract; never hand-write it. The compiler derives the 
  ‚Ėé interface from the lower layer's exported surface. You get the header's 
  ‚Ėé abstraction (a stable seam, independent rebuilds) without ever writing a 
  ‚Ėé header.

  proto-interface artifact, and docs/devices.md already states the invariant
  "the exported surface is the root set ‚ÄĒ for resolution and for reachability

  compiles against it, core compiles once).

  inlining. Arche's entire value prop is whole-column systems compiled to native
  ‚ÄĒ a dynamic seam between app and core means the column loops on each side
  can't be inlined/fused across it, plus you inherit ABI lock-in, versioning,

  - Separate compilation + static final link. Compile core (the OS/window

  Does gfx.x11.arche conflict with foo.device.arche?

  Today, no convention assigns meaning to a middle dotted segment. Inside a
  device folder the loader merges every *.arche file regardless of its base name
  (compile.c:281-289); the only filename with special meaning is the .ds.arche
  datasheet suffix (file_is_datasheet). So right now foo.device.arche is just an

  magic target tag, then foo.device.arche would suddenly be read as base foo +

  dotted segments retroactively breaks every file that uses dots descriptively.

  declared set:

  declared target, it's inert and the file always merges. So foo.device.arche
  stays a normal always-merged file unless you literally declare a target named 
  "device" ‚ÄĒ which you wouldn't.

  in the current tree changes meaning.

‚óŹ Genuinely good question to pause on ‚ÄĒ and the honest answer is:
  dependency-aversion is much stronger in compilers than in normal software, but
  it's a spectrum, not a rule, and the most admired minimal compilers 
  specifically hand-roll their config formats rather than take a parser dep.

  What real compilers actually do

  Aggressively minimal (and they hand-roll config):
  - Go ‚ÄĒ toolchain is self-contained, bootstrapped in Go, ~no external deps.
  Tellingly, go.mod is a bespoke format with a hand-written parser ‚ÄĒ they
  deliberately didn't adopt TOML/YAML, to keep parsing trivial and dep-free.
  - Zig ‚ÄĒ its build config is build.zig (config-as-code in Zig itself ‚ÄĒ the
  "arche-native" idea), plus build.zig.zon, a bespoke minimal notation they 
  parse themselves. (Zig does lean on LLVM today, but is actively building its
  own backends to drop it.)
  - TCC ‚ÄĒ essentially zero deps.

  Established, small fixed dep set (reimplement utilities, don't pull general 
  libs):
  - GCC ‚ÄĒ takes GMP/MPFR/MPC (bignum/float math) ‚ÄĒ so not zero-dep, but a tiny,
  decades-stable, carefully-chosen set.
  - LLVM/Clang ‚ÄĒ huge, but famously reimplements its own ADT/StringRef/hashmaps
  rather than pulling Boost; external deps near-zero (optional zlib).
  - CPython ‚ÄĒ core interpreter is self-contained C; deps (openssl, sqlite,
  libffi) are optional, gated stdlib extras.

  Dep-heavy (the contrast): Rust's broader toolchain pulls many crates (Cargo
  uses the toml crate) ‚ÄĒ but that's an ecosystem culture, and even rustc's core
  is more conservative.

  Why compilers trend this way

  Four reasons that genuinely apply to arche: bootstrapping (it must build from
  minimal prerequisites), portability/longevity (it runs for decades across odd
  targets; deps rot), supply-chain trust (a compiler is a trust root ‚ÄĒ
  Thompson's "Trusting Trust" makes minimal deps a security posture, not just
  taste), and reproducibility.

  What this means for your call

  You weren't enforcing zero-dep, but you've been living it, and for a compiler
  that's the mainstream, defensible default ‚ÄĒ not a quirk. The sharper point:
  for the specific thing in front of us ‚ÄĒ a tiny config file ‚ÄĒ the best-in-class
  minimal compilers (Go, Zig) concluded that a parser dependency wasn't worth
  it and rolled their own format. So "hand-roll a TOML subset" or "arche-native
  config" isn't a sacrifice you're making to be pure; it's the well-trodden
  path.

  That said, reconsidering zero-dep does unlock one option I'd previously
  down-weighted: vendoring a single small public-domain TOML parser (e.g.
  tomlc99 ‚ÄĒ one .c/.h, ~MIT) gives you full TOML for ~one file you drop in and
  link into both binaries. That's GCC-style "one small, well-scoped dep," not
  Rust-style dep-soup. It's a reasonable middle path if you'd rather not
  maintain even a 150-line parser.
