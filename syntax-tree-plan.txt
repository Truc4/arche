● Got the grounded picture. Here's the straight take.

  First, the uncomfortable part: you already built most of this

  You're reaching for an architecture you largely already have. Look at what's
  actually in the tree:

  - CST (cst/syntax_tree.h) — lossless green tree, every token + comment,
  node-id keyed. Formatter eats it directly. This is your "Source view." Already
  done.
  - SemModel (semantic/sem_model.h) — expr_type, expr_nominal, implicit_move,
  all side-tables keyed by CST node id. This is your "Semantic view." Already
  done, and it's literally annotations over the canonical CST, not a separate
  tree.
  - HIR (hir/hir.h) — separate desugared tree. This is your "Lowering view" —
  except it is NOT a view, and that's the bug in your plan.

  You independently reinvented Roslyn. Roslyn has SyntaxTree + SemanticModel.
  You have CST + SemModel. Same idea, same names almost. Your instinct is sound
  — it's just mostly shipped.

  The actual mistake in the plan: lowering is not a view

  Here's the truth you asked for. A "view" is only a view if it preserves the 
  node set bijectively. Source view, semantic view, type view, lint view, inlay
  hints, docs — all legit views, because they attach info to existing syntax
  nodes by id. One node in, zero-or-more annotations out, same nodes.

  Lowering breaks that. Your own code proves it:

  - match desugars to a synthetic __match_<id> temp + an if-chain. Those nodes 
  do not exist in source.
  - Tuples flatten: pos.(x,y) → pos_x, pos_y. New nodes.
  - Enums erase to int literals. Modules inline. Implicit moves materialize as
  UNARY_MOVE.
  - The explorer confirmed: synthetic HIR nodes carry no source location and
  have no CST id.

  That's not a view. A view can't invent nodes. The moment you desugar, you've
  changed the node set, and you're building a new tree. You already correctly
  built it as a separate tree (HIR).

  So the rule, stated cleanly:

  ▎ Does the transform preserve the node set 1:1? → it's a view (additive 
  ▎ side-table over the canonical tree).
  ▎ Does it add/remove/reshape nodes? → it's a separate IR, downstream of a hard
  ▎ boundary.

  Source and Semantic sit on the view side of that line. Lowering sits on the
  separate-tree side. Your three-bucket model is really two views + one IR, not
  three views.

  If you architect lowering as a "view over the Program Tree," you'll spend the
  rest of the project fighting desugaring — trying to maintain node
  correspondence that fundamentally cannot exist. The
  synthetic-nodes-with-no-source-loc thing is the system already telling you
  this.

  On the name "Program Tree"

  Honest: it's a downgrade, and specifically because of the above.

  - "CST" is precise — concrete = lossless, every token. The reason it grates on
  you is it sounds dumb/syntactic, when in your design it's the canonical spine
  every view hangs off. Fair complaint.
  - But "Program Tree" is vague, and worse: it invites the category error.
  "Program Tree" sounds like it should contain the whole program including the
  lowered form — which tempts you to fold HIR back in. The name pulls you toward
  the mistake.

  Better names for the canonical spine, in order: keep CST, or call it Syntax 
  Tree / Source Tree. It's the syntax. The richness comes from the views over
  it, not from renaming it to sound richer. Name the views for what they
  preserve (you're already doing this in your head: source/semantic). That's the
  good part of your plan — keep it.

  The real cruft you should actually be looking at

  Not the name. It's AstProgram + cst_to_program() (cst/cst.h). That's a legacy
  intermediate reconstructed on demand so old semantic passes keep working.
  That's the thing muddying your representation story — a third structure that's
  neither canonical tree nor view nor IR, just a transition-era ghost. If you
  want a clean model, killing AstProgram and running semantic passes directly on
  CST+views is the real win. Renaming CST buys you nothing; deleting AstProgram
  buys you a coherent pipeline.

  Scorecard

  - Canonical-tree + additive-views for Source + Semantic: right, and already 
  built.
  - "Lowering view": category error. It's HIR, a separate downstream IR. Keep
  the hard boundary.
  - "Program Tree" name: don't. Keep CST / call it Syntax Tree. The name
  shouldn't imply lowering lives inside.
  - Hidden debt: AstProgram legacy layer is what actually needs cleaning, not
  the name.

❯ how does killing it work? where does the typed view live for lowering/lsp hint gen? 

● Good question — it's the question for the next project, and the answer is
  reassuring: killing AstProgram doesn't move the typed view at all, because the
  typed view doesn't live in AstProgram today.

  Here's the actual data flow right now:

  Parser → CST (SyntaxNode, node-id keyed)
             │
             ├─ cst_to_program() → AstProgram   ← transient scaffold, ONLY
  semantic.c walks it
             │                                      (carries cst_id = node_id+1
  back to CST)
             │
  semantic.c analysis passes walk AstProgram, WRITE →  SemModel  (side-table
  keyed by node id:
                                                                    expr_type,
  expr_nominal,
                                                                    bind_alias,
  implicit_move)
             │
  Lowering:  walks CST + reads SemModel  ──────────────┐  (never touches
  AstProgram)
  LSP hints: walks CST + reads SemModel  ──────────────┘  (never touches
  AstProgram)
             │
  HIR → codegen

  The typed view IS SemModel — a side-table keyed by CST node id. Lowering
  (lower_to_hir(const SyntaxNode *root, ...)) and LSP hint gen (sem_hints, over
  the CST) already read from it directly. Neither one ever reads AstProgram.
  AstProgram is purely a walking convenience for the analysis passes that
  produce SemModel.

  So the answer to "where does the typed view live for lowering/LSP" is: exactly
  where it lives now — SemModel, keyed by node id — and the kill doesn't touch 
  that. Both consumers are downstream of AstProgram and unaffected.

  What the kill actually changes is only the input shape of the producer:

  - Today: analysis passes do Expression *e … e->data.call.args[i] … e->cst_id,
  then sem_model_set_expr_type(model, e->cst_id - 1, …).
  - After: they walk the CST via the SyntaxView accessor layer
  (sv_call_args(node), etc.) and key SemModel directly by cv_id(node) — the
  +1/-1 "unlinked" encoding disappears.

  Two distinct things, both anchored on the same CST node, neither is
  AstProgram:

  ┌─────────────────────┬──────────────────┬────────────────────────────────┐
  │       Concern       │  Where it lives  │         After the kill         │
  ├─────────────────────┼──────────────────┼────────────────────────────────┤
  │ Structure           │ SyntaxView       │ unchanged layer, extended —    │
  │ ("is-a-call, give   │ accessors (sv_*) │ add accessors for whatever     │
  │ me args")           │  over the CST    │ semantic.c currently reads off │
  │                     │                  │  AstProgram structs            │
  ├─────────────────────┼──────────────────┼────────────────────────────────┤
  │ Semantic facts      │ SemModel, keyed  │ unchanged — same table, same   │
  │ (resolved type,     │ by node id       │ keys                           │
  │ nominal, move)      │                  │                                │
  └─────────────────────┴──────────────────┴────────────────────────────────┘

  That's the rust-analyzer shape exactly: one syntax tree + typed accessors + a
  type side-table; no parallel AST. The clean end-state mental model:

  - CST = canonical structure
  - SyntaxView (sv_) = typed structural lens over it
  - SemModel = semantic side-table (your "Semantic view")
  - Lowering & LSP = walk CST via SyntaxView + query SemModel by node id —
  identical pattern, AstProgram absent.

  So the kill = three mechanical moves: (1) rewrite the ~1,000
  ->data./typed-pointer sites in semantic.c to walk SyntaxView instead, (2)
  delete cst_to_program (~560 lines) + the AstProgram/Decl/Statement/Expression
  structs + cst_id fields, (3) extend the SyntaxView accessor layer wherever a
  needed lens doesn't exist yet. The output contract (SemModel) and both
  consumers (lowering, LSP) stay put.

  Let me capture this design sketch in the plan so it's not lost when we get to
  it.
