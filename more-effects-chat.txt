в—Ҹ All three lenses are in, and they cohere into one answer. Here's the
  synthesis.

  Your loophole is real, and it points at the right fix

  Both design lenses independently concluded: the procвҶ’proc ban is only real if
  wrap cannot ingest anything that runs an effect. So the rule is:

  в–Һ You can wrap an extern (a foreign leaf with no arche body). You cannot wrap 
  в–Һ a proc.

  Your 3-step recipe dies at step 2: wrap_P :: func(fd) -> Eff(bool) { return 
  wrap P; } simply doesn't typecheck, because P is a user proc, not an extern.
  There is no expression that turns a proc into an Eff. That's what makes
  "unspellable" literally true instead of an aspiration.

  "Can any proc be wrapped?" вҖ” No, and here's the principled line

  An extern is inert without its outputs: "the op minus its out-slots" is just
  data вҖ” an opcode plus its inputs. That's a legitimate value (Eff).

  A proc is a body that runs effects. "The proc minus its outputs" is not data вҖ”
  it's a suspended computation, which is exactly the monadic thunk the model
  refuses to have as a value (the free-applicative-not-monad line; the depth-1
  spine). So a proc can't become an Eff because an Eff is inert and a suspended
  proc isn't.

  That ontological split вҖ” inert foreign leaf vs. composite runner вҖ” is the
  whole game. Wrapping a leaf keeps the spine system вҶ’ proc вҶ’ extern at depth-1.
  "Wrapping" a proc would make it system вҶ’ Q вҶ’ P вҶ’ extern = depth-2,
  recursively unbounded.

  The syntax that makes this a theorem instead of a rule (saturation)

  The syntax lens's recommendation, which I think is the strongest result of the
  three: drop the wrap keyword entirely and define Eff by saturation.

  Eff(TвҖҰ) is an extern under-applied by its out-slots. Supplying the out-slots
  runs it. One rule for the whole language:

  The bonus finding вҖ” fix the doc's justification, not just the loophole

  So В§5/В§9 should re-found the ban on legibility + inertness, and demote the

  The one honest cost (unchanged, now sharper)

  Result-dependent reuse has no home: a "read header вҶ’ branch on length вҶ’ read
  body" or a retry loop is monadic (can't be a func) and not an extern (can't be

вқҜ why wrap? what is the point in providing the in and out params at different   
  times?                                                                        

в—Ҹ Good question вҖ” and it splits into two that the doc currently conflates: is 
  the in/out split real? and does wrap (the keyword) earn its place? Different
  answers.

  In-params and out-params aren't symmetric вҖ” that's the whole reason

  The split isn't a timing convenience. It mirrors the actual structure of an
  effect:

  - In-params are causes. You compute them вҖ” the fd, the formatted row, the
  length. Pure work. Known before anything happens.
  - Out-params are results. You observe them вҖ” the bytes written, the errno,
  what the peer sent. They don't exist until the world runs the effect.

  So they're available at different times because they are different times: you
  author the request; the world authors the response. Eff(int, int) = "request
  fully specified, response not yet produced." That's exactly Haskell's IO a вҖ”
  the a is the not-yet-known result, the inputs are closed over. Baking the
  inputs in is what makes the thing a complete, self-contained pending action
  rather than a half-applied function pointer.

  And that completeness is the only reason effects can be values:

  eff := over_budget ? log_warn(msg) : log_ok(msg);   // pure: pick WHICH
  effect; inputs already baked
  eff(ok:);                                            // impure: run the chosen
  one, once

  // build a column of pending writes in a pure map; run them all in one impure
  each-kernel
  rows  :: map query { line } { line = fmt(price, qty); }   // pure: compute
  requests
  flush :: each query { line, ok } (fd) { write_b(fd, line)(n:, err:); ok =
  err==0; }  // impure: run

  If inputs were supplied at run time too, an "effect value" would be incomplete
  вҖ” you couldn't choose among them, store them, or collect them, because the
  value wouldn't contain what it's going to do. The free-applicative layer (В§4),
  the each batching, the testability ("the func that builds the request is pure
  вҶ’ no mocks") вҖ” all of it rests on the inputs being in the value. The split is
  load-bearing.

  But wrap the keyword is ceremony вҖ” drop it

  Here's the part the doc gets wrong. Given the split, you do not need a keyword
  to construct it. An extern call with its out-slots omitted is already "inputs
  given, outputs owed" вҖ” that is the Eff:

  write_b :: func(fd, buf) -> Eff(int,int) { return fwrite(fd, buf); }  //
  under-applied = the Eff
  ...
  write_b(fd, line)(n:, err:);                                          //
  out-slots supplied = run it

  wrap adds nothing the arity doesn't already say. So: keep the in/out split
  (it's the definition of an effect value), delete the keyword (saturation gives
  it for free вҖ” this is the synthesis from the syntax lens).

  And when the split earns nothing, don't force it

  If you're doing one immediate effect and never treat it as data вҖ” no choosing,
  no collecting, no pure/impure seam вҖ” then the split is pure overhead and the
  language should let you just write the saturated call in the proc:

  fwrite(fd, buf)(n:, err:);   // inputs and outputs at once вҖ” no Eff value ever
  materializes

  That's the same syntax as running a built Eff; it just happens to
  construct-and-run in one breath. The value world is opt-in: you reach for the
  deferred form precisely when you want to choose, batch, store, or test вҖ” and
  you pay for it only then.

  Bottom line: the in/out split is real and definitional (causes vs. results;
  it's what makes an effect a value). wrap is not вҖ” it's a keyword for a
  distinction the call's arity already encodes. The doc should justify the split
  on the cause/result asymmetry, not present wrap as the mechanism.

вқҜ can a func collapse a series of procs into a single eff?                      

в—Ҹ Three different questions are hiding in that one, and they have different
  answers.

  "A series of procs" вҶ’ No, never

  A proc isn't a value. A func can't take a proc as an argument, can't name one
  as data, can't bundle it. (Same rule that closes the loophole: a proc minus
  its out-slots is a suspended computation, not an inert Eff.) So there is no
  func that ingests P, Q, R and emits an Eff. That construct doesn't exist by
  design вҖ” it's exactly what would make procs first-class and break flatness.

  "A series of effects, statically ordered" вҶ’ Yes вҖ” this is the point of the 
  func world

  If you mean "compose several externs into one Eff," yes, and it's the intended
  answer to "how do two procs share an effectful step?" A func composes effect
  values, and the composition is pure вҖ” it nests freely:

  preamble :: func(fd) -> Eff(int,int,int) {
    return seq(fwrite(fd, "id,price,qty\n"),   // вҶҗ order is fixed at build
  time,
               fwrite(fd, "---\n"),            //    no result feeds the next
  input
               fwrite(fd, "begin\n"));
  }
  write_head :: proc(fd)(ok) { preamble(fd)(n1:,e1:, n2:,e2:, n3:,e3:); ok =
  (e1|e2|e3)==0; }

  The single Eff carries the union of out-slots; running it runs all three
  externs in order. Crucially, the spine stays depth-1. Collapsing adds breadth,
  not depth: system вҶ’ proc вҶ’ {extern, extern, extern}. The composition happened
  in the value world ("just math," which nests for free); when run, it flattens
  to a sequence of terminal externs with no proc in between. So this collapse
  is fully legal and doesn't dent the bound.

  "A series where step N depends on step <N's results" вҶ’ No вҖ” that's a monad, 
  and it stays in the proc

  The moment a later effect's input is an earlier effect's output, you need to
  observe a result to build the next request. A func can't вҖ” it never runs
  anything. That sequencing is monadic bind, and the value layer is deliberately
  a free applicative, not a free monad (a free monad would have to carry
  runtime continuations = allocation + unbounded = exactly what arche refuses).
  So it can only be expressed imperatively, inside one proc:

    len := int(hdr[0]);                 // result of the first effectвҖҰ
    read(fd, body, len)(got:, err:);    // вҖҰbecomes the input of the second вҶ’

вқҜ so what happens when everything becomes arche native. It seems there is no    
  way to create effects without extern.                                         

в—Ҹ Right вҖ” and that's not a gap, it's the most honest thing about the model. But
  your premise is slightly off in one important way, and fixing it answers the
  question.

  You can never create an effect in arche source вҖ” by design

  An effect is an interaction with something outside the pure language. There is
  no such thing as an effect that doesn't, at the bottom, touch the world. So
  no pure code can ever conjure one вҖ” it can only name a primitive one and then
  transform/compose/choose it. This isn't arche being limited; it's universal:

  - Haskell: there is no pure-Haskell function that builds a new IO from
  nothing. Every actual IO effect is an RTS primitive or FFI (putStr,
  readMutVar#, foreign import). pure/return injects a value, not an effect.
  - Rust: every I/O bottoms out in unsafe + a syscall/intrinsic.

  So "the only origin of an effect is a primitive at the boundary" is correct
  and load-bearing: it's what makes the effect set closed and auditable вҖ” you
  can list every atomic effect a program can perform by listing its primitive
  blocks. That property is the prize. Native doesn't threaten it.

  But it isn't only extern вҖ” arche already has a native effect primitive

  insert / delete on pools are effects, and they're not foreign. They're
  built-in primitives: they mutate the world (the columns), routed through the
  deferred command buffer and flushed at a stage barrier. The doc even says the
  whole effect model is that pattern generalized. So the real statement is:

  в–Һ Effects come from primitives. extern (foreign C) is one category of 
  в–Һ primitive. Structural mutation (insert/delete) is another, and it's 
  в–Һ pure-arche-native.

  Pure funcs can't invent atomic effects; they compose the primitive set. That
  set is fixed by the platform, not by user code.

  So what "everything native" actually does: it shrinks and sharpens the 
  boundary

  "Native" is two different things, and only one of them ever involved effects:

  - Native logic (parsing, math, sorting, data structures) вҖ” was never an 
  effect. As arche reimplements libc's strtol, qsort, JSON, etc. in arche, they
  become funcs. Most of "going native" is just more pure code. The effect
  surface doesn't grow вҖ” it's untouched.
  - Native I/O (a TCP stack, a filesystem, an allocator in arche) вҖ” even these
  bottom out at hardware: a NIC register poke, an MMIO write, a syscall trap, an
  interrupt. Those irreducible hardware touches are still primitives. You write
  the TCP logic as funcs; the "store to this device register" is the leaf.

  The interesting consequence: today extern fwrite hides a pile of libc
  buffering + a syscall behind one opaque leaf. Go native and you extern write
  the syscall itself, and all the buffering/formatting libc did becomes arche
  funcs. So native pushes effects down to their true atoms and pulls everything 
  above them into the pure world. The auditable effect list gets smaller and 
  more honest, not bigger.

  The design implication: generalize "extern" to "primitive"

  extern proc is currently spelled as "foreign C function," but the concept is
  "an irreducible world-boundary leaf arche can't express in arche." A
  freestanding/self-hosted arche wants sibling surfaces for the same concept:

  #foreign {
    // category 1: foreign function (C ABI) вҖ” what exists today
    write :: extern proc(fd: int, buf: []char, n: int)(ret: int, err: int)
  }
  #primitive {
    // category 2: the real hardware atoms a native arche needs
    syscall :: prim proc(nr: int, a: int, b: int, c: int)(ret: int)        //
  trap to kernel
    mmio_w  :: prim proc(addr: int, val: int)()                            //
  volatile device store
    rdtsc   :: prim proc()(cycles: int)                                    //
  asm intrinsic
  }

  Every one of these is the same thing the model already wraps: an inert,
  world-touching leaf you compose into Eff and run in a proc. The model doesn't 
  change вҖ” only the primitive vocabulary grows. And the invariant survives
  intact: pure code still can't create an atomic effect; the closed set of
  primitives still is the program's complete effect surface, only now it's the
  genuine syscall/MMIO boundary instead of "whatever libc happened to do."

  So: a fully-native arche still has effects, they still all originate at
  primitives (foreign call, syscall, MMIO, asm, or structural mutation), and
  pure code still can't manufacture one. That's exactly the property you want вҖ”
  going native makes the boundary thinner and the effect audit truer, it doesn't
  make effects disappear.

  The convenience layer becomes funcs that return Eff вҖ” and it stays in the lib

  The rule falls straight out of the applicative/monadic line. Almost every
  convenience wrapper is pure shaping + a (statically-composed) primitive, and

  two ways, both legal in the value world:

  leaf is always the real primitive (net_send, net_recv, os.syscall), never
