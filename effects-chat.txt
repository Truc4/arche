вқҜ so if effect descriptions are a thing. I don't think procs serve a purpose.   
  They have been supersceded by handlers, no?                                   

в—Ҹ Short answer: no. Handler is proc's dual, not its replacement.

  The duality. Every performed effect needs two ends:
  - initiator вҖ” someone performs (kicks it off). Active, scheduled, sequenced by
  run.
  - reactor вҖ” someone handles (catches + acts + resume). Passive, installed,
  waits to be routed to.

  proc = call-once initiator. handler = reactor. Doc says this directly: handler
  is "the reactive dual of a proc (the same leaf body bound to a run/schedule
  site, which initiates instead of reacting)." Delete procs вҶ’ nothing starts the
  effect. Handlers never run on their own; they only fire when something
  performs.

  Effect-descriptions don't change this. Description is inert data. Chain is:
  func builds description вҶ’ proc PERFORMS it вҶ’ handler does the work
  func can build but never perform вҖ” perform is impure, func is pure. So the
  perform must live in an impure call-once kind. That kind is proc. Descriptions
  kill proc-as-plumber (forwarding data between procs вҖ” now func's job, that's
  why procвҶ’proc banned). They don't kill proc-as-performer.

  Grain of truth вҖ” worth naming. Your instinct bites on the thin one-shots:
  step_clear :: proc()() { clear(...) }, step_present, respond_text. Those exist
  only to wrap one perform so a system can run it. Real question hiding here:

  в–Һ can the initiator role be absorbed вҖ” system or schedule performs inline вҖ” 
  в–Һ collapsing proc+handler into ONE "effect leaf" kind, distinguished only by 
  в–Һ trigger (schedule-triggered vs perform-triggered)?

  That's coherent (it's the "everything is a handler, the schedule is the only
  performer" design). But two costs:
  - breaks В§5 firewall "no construct both performs and installs" вҖ” system
  installs, so it can't perform; that firewall is exactly what keeps proc
  separate. Relax it вҶ’ pyramid-of-with nesting returns.
  - it's a rename/unify, not a supersession вҖ” the schedule-triggered "handler"
  does precisely what the proc did. Initiator role is irreducible; you'd just
  call it something else, and lose the clean initiate-vs-react split + the
  system param-threading model.

  So: handlers вү  procs. The thing that could eat thin procs is the
  system/schedule, not the handler вҖ” and the design deliberately blocks it.

  Want me to record this as a В§9 open question ("can the call-once performer
  collapse into system/schedule, unifying proc+handler") + a sharper note in В§5
  that proc/handler are duals not substitutes?

вқҜ yeah but systems compose handlers and functions (which return effect          
  descriptions) or does that not work and we have to use procs to "use" the     
  effect                                                                        

в—Ҹ Right question. Answer: system can compose handlers + funcs, but it still 
  can't perform вҖ” so the description stays inert until a leaf uses it. That leaf
  is the proc (or each).

  Walk the mechanics:

  func   builds  the description   (pure вҖ” returns Emit("%d", x), performs
  nothing)
  proc   PERFORMS the description   (impure, call-once вҖ” the "use")
  handler does    the work          (impure, terminal вҖ” renders bytes, os.write)
  system INSTALLS handler + wires   (no compute, no perform)

  System has handle + run + thread. It does not have perform. Blocked by the В§5
  firewall: "no construct both performs and installs." So a system holding a
  func's returned Emit value can't turn it into a real effect вҖ” it can only hand
  it to a leaf that performs.

  Why the firewall, not arbitrary:
  1. Pyramid-of-with. Install lives in a flat system preamble; perform lives in
  a flat leaf body. Let one body do both вҶ’ handler-installing-bodies nest вҶ’ the
  OCaml/Haskell with-pyramid the design kills.
  2. Untracked effects. If "system uses func's return value" silently performed,
  a pure-looking func call would cause effects when consumed. That's exactly
  the OCaml-5 untracked-effect footgun В§5 rejects. Perform must be a visible act
  in an impure kind.

  So your two-turn arc lands here:

  в–Һ proc is not superseded. It's the call-once performer вҖ” the only thing that 
  в–Һ can "use" a description at call-once grain, because func can't (pure) and 
  в–Һ system can't (firewalled).

  One refinement to your phrasing: the irreducible category is "performing 
  leaf," not "proc" specifically. proc = call-once member, each = fanned member.
  A system with only run each/run map needs zero procs вҖ” but it still needs a
  performer. You never escape needing one; you just pick call-once (proc) vs
  fanned (each).

  Net: func + handler + system is not enough. The perform is a hole only a
  proc/each fills. That's proc's whole reason to exist post-descriptions вҖ” not
  plumbing (descriptions killed that), but being the visible, impure, call-once 
  perform site.

  Want the doc note now? I'd add to В§5: "proc/handler/system are three 
  irreducible roles, not substitutes вҖ” perform (proc/each) вҹӮ react (handler) вҹӮ
  install+wire (system); the install/perform firewall is what forbids the system
  from absorbing the perform." Plus drop the В§9 "can the performer collapse"
  question, since this converged to no.

  across handler scopes вҖ” build it under one installed handler, perform it under
  another. That power is dead here вҖ” the handler is fixed statically by where

  For a proc, which is already impure and performs now, returning is pure
  ceremony. Perform is the honest primitive.

  "I want procs to do multiple effects" вҖ” yes, that's already the model, and 
  it's the right center. A proc is a flat sequence of performs, depth 1:

  one-effect. The single-effect limit (return AN effect, first-order, no monadic

вқҜ what is free applicative                                                      

в—Ҹ Applicative vs monad вҖ” the one distinction that matters here:

  - Monad (>>=): a later effect can depend on an earlier effect's value.
  n    <- read_header        // get a value
  body <- read_bytes(n)      // the NEXT effect's shape depends on n
  - The computation's structure isn't known until you run it вҖ” read_bytes(n)
  doesn't exist until n does.
  - Applicative (<*>): you can combine independent effects, but none can depend 
  on another's runtime value.
  combine(+)  read_x  read_y    // both reads fixed up front; (+) applied to
  results
  - The structure is fully known before execution вҖ” read_x and read_y are both
  there from the start, no value flows between them.

  That "structure known up front" is the whole property. It's why independent
  effects can be inspected, reordered, batched, run in parallel вҖ” there's no
  data dependency to respect.

  "Free" = the data-structure version. A free applicative is: take your set of
  operations (your effect descriptors), and get вҖ” for free, mechanically вҖ” the
  ability to combine them into a fixed plan, without giving them meaning yet.
  The plan is just data: a list of operation-nodes plus a pure function to fold
  their results. Meaning comes later, from one interpreter that walks the
  structure and actually does the work.

  This is your design, exactly:
  - effect descriptor = one operation node (the functor)
  - the command pool (a list of descriptors, all known before draining) = the
  free applicative's fixed structure
  - executor proc / runtime executor = the interpreter (the natural
  transformation to real I/O)
  - the thing you can't do вҖ” read_header then read_bytes(n) per row вҖ” = bind,
  the monad upgrade you don't have

  So when the Haskell lens said "free applicative in denial," it meant: your
  command-pool-plus-executor is the textbook Ap f construction. And that's good 
  news, two ways:

  1. Your parallelism is sound by theorem, not luck. "Applicative effects have
  no inter-effect data dependency" is exactly the proof that your cores-backend
  can run stages/rows in parallel. You get to cite it instead of asserting it.
  (Same reason optparse-applicative can build --help by inspecting all flags
  before parsing, and Haxl batches independent fetches.)
  2. Your one hole has a precise name and boundary. The free applicative
  provably cannot express result-dependent sequencing. So "read a length, then
  read that many bytes" isn't a vague gap вҖ” it's the exact ApвҶ’Free
  (applicativeвҶ’monad) boundary. You either climb to a bounded monad for that one
  case, or declare it out of scope. Either way it's a known line, not a
  surprise bug.

  The lattice, top to bottom: Functor (map a result) вҶ’ Applicative (combine
  independent effects, static structure) вҶ’ Monad (effects depend on prior
  results, dynamic structure). You're parked at the applicative rung on purpose.
  The value: static inspectability + parallel scheduling. The cost: no value
  can flow from one effect into the shape of the next.
