# E0225 — `main` is reserved

A program's entry point is a **`#run`** schedule, never a decl named `main`. So `main` carries no special
meaning and is reserved: a user decl called `main` — of any callable kind (`func`, `system`, `each`,
`map`, and the now-foreign-only `proc`) — is rejected, so nothing can masquerade as an implicit entry.

There is no implicit "the `main` function runs first." An executable is its declarations plus exactly one
`#run <schedule>`; the runtime executes that schedule. Setup is just an entry before a loop in a `seq`;
the per-tick work is the loop body. (Foreign C symbols named `main` are unaffected — this is about
arche-authored decls.)

## Example

```arche
// BAD: a decl named `main` — E0225
main :: system {
  fmt.printf("hi\n");
}
#run main

// GOOD: name it anything; `#run` is the entry
entry :: system {
  fmt.printf("hi\n");
}
#run entry
```

For a setup-then-loop program:

```arche
#run seq({ boot, forever(frame) })   // `boot` runs once, then `frame` loops — no `main` anywhere
```

## How to fix

Rename the decl (e.g. `entry`, `boot`, `serve`) and name it in `#run`. If several things must run in
order, compose them with the schedule combinators (`seq`, `forever`, …), not a top-level `main` that calls
them — composition lives in the `#run` schedule.

See `docs/design/scheduling.md` and `docs/design/the-flat-effect-model.md` §9.
