# W0027 — pointless move

A `move` that transfers nothing is reported. `move` exists to hand off **ownership** of a
value — after `move x`, the source `x` is consumed and may not be used again. Applying it to
something that has no ownership to give up is a no-op that reads as if it did something.

This is an ordinary lint: **warning by default**, tunable (see below).

## What trips it

`move` applied to a **pool column** — either a bare column or a column slice:

```arche
b :: char;
Buf :: arche { b }
[65536]Buf;

io.read_chunk(fd, move Buf.b[0:Buf.capacity], 65536)(chunk:);  // W0027 — drop `move`
```

## Why it's pointless

A pool is **static, shared storage** — there is no heap, nothing is ever freed, and a column is
owned by the pool for the whole program. A column slice (`Buf.b[0:Buf.capacity]`) is therefore just
a **borrow** of that stable storage: a `{ptr, len}` view. There is no ownership to transfer, so
`move` changes nothing.

It also consumes nothing: a column slice is an *expression*, not a variable, so unlike `move x`
(which marks the variable `x` consumed) there is no binding for `move` to retire. The keyword is pure
noise here — and worse, it suggests a transfer that isn't happening.

## How to fix or opt out

- **Drop the `move`** — pass the column (slice) as a plain borrow. Readers and IO that take a
  `buf: []T` borrow write through it just the same:
  ```arche
  io.read_chunk(fd, Buf.b[0:Buf.capacity], 65536)(chunk:);
  ```
- **Suppress per declaration** with `@allow(pointless_move)` on the enclosing proc/func.
- **Promote to an error** with bare `-Werror`.

## Not flagged

- `move x` of an owned local/parameter (a real ownership transfer).
- A pool column passed as a **borrow** (no `move`) — the blessed way to use it.
