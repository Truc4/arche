# E0210 — non-exhaustive match

A `match` on an enum or integer must cover every case. Arche checks this at
compile time: either list an arm for every enum variant, or add a `_` catch-all
arm. A match that can fall through with no arm taken is a bug, so it is rejected.

## Example (rejected)

```arche
Method :: enum { get, post, delete }

handle :: proc(m: Method)() {
  match m {
    get  : printf("GET\n");
    post : printf("POST\n");
    // missing `delete` — and no `_` — so this is non-exhaustive
  }
}
```

```
error[E0210]: non-exhaustive match: missing 'delete' (cover every variant, or add a `_` arm)
```

## Fixes

Cover the missing variant:

```arche
match m {
  get    : printf("GET\n");
  post   : printf("POST\n");
  delete : printf("DELETE\n");
}
```

…or add a catch-all, which also handles any future variants and integer matches
(where the full set isn't enumerable):

```arche
match m {
  get : printf("GET\n");
  _   : printf("other\n");
}
```

## Why

`match` desugars to a direct-dispatch if-chain. Exhaustiveness is what makes that
dispatch total — every input lands on exactly one arm — so behavior is never
silently undefined when a case is forgotten or a variant is later added.
