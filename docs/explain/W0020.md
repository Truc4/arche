# W0020 — redundant guard

A guard-exit at the **start** of a loop body re-tests the loop's **own condition** —
so it can never fire. A `for` loop re-checks its condition at the top of every
iteration, so that condition already holds for the whole body; an `if` that tests
its exact complement on the same operands is dead control flow.

## Example

```arche
for (i := 0; i < s.length; i = i + 1) {
  if (i >= s.length) {   // W0020: the loop already guarantees `i < s.length` here
    break;
  }
  acc = acc + s[i];
}
```

## How to silence

- **Delete the guard** — the loop condition already covers it.
- **`@allow(redundant_guard)`** on the enclosing decl.

## Not flagged (deliberately conservative — no false positives)

The lint only fires when the prover can be *sure* the operands still hold their
loop-top values, so it covers a narrow, sound case:

- **Only a *leading* guard** (one in the run of guard-exits before the first other
  statement). A guard-exit only exits or falls through unchanged, so until some
  other statement runs, the loop variable and bound are still the loop's. A guard
  *after* an assignment is left alone — that statement may have changed an operand.
- **Only the exact complement on the same operands** — loop `<` ↔ guard `>=`,
  `<=` ↔ `>`, `>` ↔ `<=`, `>=` ↔ `<`, with the same variable and the same bound
  (a name, an int literal, or the same `.length`/`.cap` extent). A guard on a
  *different* bound (`if (i >= limit)` under `for (; i < n; …)`) is real control flow.
- A guard whose bound the prover can't match exactly (computed indices, mixed
  expressions) is never flagged — it is not a guess.

This is **not** a general dead-branch analysis: it does not reason about value
ranges across iterations (e.g. `if (k < 0)` where `k` is provably non-negative by
arithmetic) — only the syntactic "you already wrote this in the loop header" case,
which is sound regardless of how the body mutates the operands later.
