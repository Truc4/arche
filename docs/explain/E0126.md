# E0126 — `!undefined` forbidden

`!undefined` is the **raw, unchecked** failure policy: it opts out of *all* runtime safety, emitting a
bare memory op with no bounds check (undefined behavior if the index is actually out of range). Because
it is the one genuinely unsafe op in the language, it is **forbidden by default** — an ordinary build
rejects it.

## Example

```arche
entry :: system {
  a: [4]int;
  i := 9;
  x := a[i] !undefined;   // E0126 — raw, unchecked access
}
#run entry
```

## How to fix

- **Use a checked total policy** — `a[i] !clamp` (or `!wrap`, `!zero` for a divide). It stays total, so
  it can never read out of bounds.
- **Make the access provably in bounds** — a literal index into a sized array, or guard it
  (`if (i < a.length) { … }`) so the bounds prover discharges it and no policy is needed.
- **Opt in deliberately** — pass `--allow-undefined` for a trusted/embedded build where you accept raw
  ops (`--unchecked` implies it). This is the escape hatch, off by default.

## Notes

- The bundled `core`/`stdlib` are exempt — only your own code is checked.
- `!undefined` is **never** allowed inside a `policy` body, even with `--allow-undefined`: a policy is
  the safety mechanism and must stay total ([E0213](E0213.md)).
- This replaces the old opt-*out* model (a `--no-undefined` flag): the default is now safe, and you opt
  *in* to the unsafe op instead of opting out of it.
