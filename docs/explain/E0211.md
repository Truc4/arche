# E0211 — proc/func types cannot be archetype components

An archetype describes **data** — columns and meta fields laid out for bulk,
cache-friendly iteration. A proc/func type is **behavior**, not data, so it
cannot be a component of an archetype (inline or via a named callable alias).

## Example (rejected)

```arche
on_hit :: proc()(dmg: int)        // a proc TYPE

Entity :: arche {
  hp     : int,
  on_hit : on_hit,                // ← behavior in a data record
}
```

```
error[E0211]: proc/func types cannot be archetype components ('on_hit') — archetypes are data; dispatch with `match` or a map
```

## Why

Per-row function pointers in a hot archetype are the classic data-oriented
anti-pattern: they defeat the straight-line, branch-predictable iteration the
layout exists to enable, and they can't be statically guaranteed non-null. Arche
has no runtime proc values at all, so there is nothing to store here.

## What to do instead

- **A tag column + `match`** — store an `enum` discriminant per row and dispatch
  on it; the compiler lowers `match` to a direct-call chain (no pointers):

  ```arche
  Kind :: enum { goblin, dragon }

  Entity :: arche { hp: int, kind: Kind }

  hit :: map (hp, kind) {
    match kind {
      goblin : hp = hp - 1;
      dragon : hp = hp - 10;
    }
  }
  ```

- **A map** (`map`) that encodes the behavior over the whole archetype.

- **Behavior as data** — route the cases into pools and let a consumer system/`each`
  react per row (the producer/consumer pattern, `docs/design/callbacks-as-data.md`),
  rather than passing a function value.
