# E0214 — cyclic policy

A `policy` applies itself, directly or transitively. A policy is a **macro**: it is inlined at every
fallible op that selects it. So a policy whose body applies itself — `a[i] !self`, or `a !p` where `p`
eventually applies the first policy again — would inline forever, overflowing the compiler's stack at
codegen. It is rejected at semantic time instead.

## Examples

Direct self-reference:

```arche
@policy (bounds) evil :: policy (len: int, i: int) {
  buf: [3]int;
  i = buf[i] !evil;   // E0214 — applies itself
}
```

Mutual cycle:

```arche
@policy (bounds) pa :: policy (len: int, i: int) { c: [3]int; i = c[i] !pb; }  // E0214
@policy (bounds) pb :: policy (len: int, i: int) { c: [3]int; i = c[i] !pa; }  // E0214
```

## How to fix

- **Drop the self-referential `!policy`** on the access inside the body.
- **Make that access provably in bounds** (a literal index, or guard it) so it needs no policy at all.
- **Use a different total policy** that doesn't lead back to this one (`!clamp`, `!wrap`).

## Not flagged

- **Acyclic nesting** — a policy applying a *different* policy that doesn't lead back is fine
  (`table[i] !clamp` inside an eviction handler).
- **The implicit default** — an unannotated fallible op in a policy resolves to a core *total* policy
  (`clamp`/`zero`) that doesn't index, so it can never close a cycle. Only explicit `!P` applications
  are considered.
