# W0016 — an `insert` into a `reject` pool ignores `ok`

A pool with no overflow policy (or an explicit `?reject`) is **fallible**: when it is full, `insert` does
not abort — it drops the row and reports `ok = 0`. Discarding that `ok` with `_`
(`insert(P{…})(_:, _:)`) silently ignores a real failure: the row you thought you inserted may not exist.

W0016 is **error-by-default** for exactly this case. It fires only when the resolved overflow policy is
`reject` (the baseline) — an infallible policy never reaches it.

## The two fixes

**Handle `ok`** — bind it to a real name and branch on it:

```arche
[16]Pending;   // no policy → reject (fallible)

enqueue :: system {
  insert(Pending { job: j })(_:, ok:);
  if (!ok) { drop_or_retry(); }
}
```

**Or declare an infallible overflow policy** on the pool, so there is no `ok` to handle — overflow is
handled for you:

```arche
[16]Pending ?abort;        // a full pool is a bug → crash loudly (no `ok` needed)
[16]Pending ?evict_oldest; // … or make room by evicting the oldest row
```

With an infallible policy, `insert(P{…})` (no out-list) and `insert(P{…})(h:)` (handle only) are the clean
call forms.

## Resolution order

The overflow policy is resolved per-call `?name` › the pool's `[N]P ?name` › `@default(proc, pool, X)` ›
the `reject` baseline. `@default(proc, pool, abort)` makes every policy-less pool crash-on-full
program-wide (so they stop requiring `ok`).

## Tuning

- `@allow(discarded_ok)` on the enclosing decl suppresses it at one site.
- `--discarded-ok=warn` demotes it to a warning; `--discarded-ok=allow` silences it; `--discarded-ok=error`
  is the default.
