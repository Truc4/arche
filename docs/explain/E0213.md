# E0213 — policy uses `!undefined`

A `policy` body used `!undefined` — the **empty/raw** failure policy that opts out of *all* runtime
safety, leaving a bare, unchecked op (UB on an out-of-bounds index or a divide-by-zero).

A policy **is** the safety mechanism: it is inlined at *every* fallible op that selects it, so a policy
that can read out of bounds silently undermines safety wherever it is used. A policy must therefore stay
**total** — every fallible op in its body must be either provably in bounds or resolved by a *checked*
total policy.

This is independent of `--no-undefined` (which bans `!undefined` in ordinary user code): inside a
policy, `!undefined` is **always** an error.

## Example

```arche
@policy (bounds) leaky :: policy (len: int, i: int) {
  z := table[i] !undefined;   // E0213 — raw, unchecked read inside a policy
  i = z % len;
}
```

## How to fix

- **Use a checked total policy** on the access: `table[i] !clamp` / `!wrap` (or `!zero` for a divide).
- **Make the access provably in bounds** — a literal index, or guard it
  (`if (i < table.length) { … }`) so the bounds prover can discharge it.

## Allowed (not flagged)

- An **unprovable but total** access — e.g. a pool overflow handler that scans a column with the
  implicit total default (`Foo.ts[j]`, which clamps). The index need not be statically provable; it
  just may not be *raw*.
- Mutating the policy's operands, arithmetic, branches, and calling the terminate/diagnostic
  primitives (`_exit`, `dprintf`) — an `abort`-style policy terminates deliberately, which is not UB.

## Related

- [E0098](E0098.md) — `!abort` inside a func/policy (a policy may not abort via the `!abort` op either;
  an `abort` *policy* terminates by calling `_exit`).
- A user-defined **empty** policy (`@policy(bounds) raw :: policy(len,i){}`) is `!undefined` by another
  name. Using it inside another policy is a deliberate raw opt-out and is not caught by E0213.
