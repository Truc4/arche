# W0024 — a system cannot write to a foreign pool

A `sys` (system) iterates the rows of ONE archetype — its per-entity columns. It may **read** another
pool as shared context (a singleton: a `[1]` pool the driver owns), but it may **not write** to a pool it
does not iterate.

The reason is the data-parallel system model: each `column = expr` assignment auto-loops over the iterated
archetype's rows. A statement that writes a *foreign* pool isn't tied to that iteration, so it would run
**once per `run`, not once per entity** — almost always a mistaken per-entity reduction (e.g. trying to
sum a value across all entities into a singleton). Rather than silently produce that wrong result, the
compiler rejects it.

The rule: a system **READS** shared singletons; the **driver WRITES** them.

W0024 is a lint that is **error-by-default** (see `--explain` note below). It is the dedicated guard
against the silent run-once footgun, so it fails the build like a hard error — but unlike a hard error it
is tunable, because the run-once write is occasionally what you want (a deliberate single-shot side effect
from a system that you know iterates one row).

## Example

```text
gravity :: int
v       :: int
Config :: arche { gravity }
Thing  :: arche { v }
[1]Config
[2]Thing

bump :: sys (v) {
  Config.gravity[0] = v;   // W0024: a system cannot write to 'Config'
}
```

## Fix

- **Reading shared context is fine** — keep the singleton read, drop the write:
  ```text
  bump :: sys (v) { v = v + Config.gravity[0]; }   // OK: reads the singleton, writes its own column
  ```
- **Set the singleton from the driver**, which owns it:
  ```text
  insert(Config, 9)(_:, _:);   // create + set the one instance (driver side)
  Config.gravity[0] = 9;       // or update it later (driver side)
  ```
- **Per-entity accumulation into a singleton** (sum/count across entities) is not expressible in a system
  today — there is no unified per-row body loop. Do the reduction in the driver after the `run`, reading
  the entities' columns.

## Opting out

W0024 is error-by-default but tunable:

- `--sys-foreign-write=warn` — demote to a warning (the write still compiles).
- `--sys-foreign-write=allow` — silence it entirely.
- `--sys-foreign-write=error` — the default (explicit form).
- `@allow(sys_writes_foreign_pool)` on the enclosing declaration — a localized escape hatch.

If you opt out, remember the semantics: the foreign-pool write runs **once per `run`**, not per row. The
opt-out is meaningful only for a **row-independent** single-shot write (e.g. `Config.gravity[0] = 7`). A
foreign write whose RHS reads an *iterated* column (e.g. `Config.gravity[0] = v`) mixes two loop domains —
the singleton's single row and the iterated archetype's many rows — which is inherently ambiguous (which
row's `v`?) and will not codegen even under `--sys-foreign-write=allow`. Do that reduction in the driver
after the `run`.
