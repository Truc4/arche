# E0047 — a collective cannot appear in a `map` body

`reduce`, `scan`, and `sort` are **collectives**: whole-column operations. A `map` is the
opposite — a per-element kernel that sees one element at a time and never the whole column. So a
collective inside a `map` body is a category error and is rejected here.

```text
// not:
bad :: map (v, s) {
  s = reduce(+, v);   // a whole-column fold inside a per-element kernel
}
```

## Instead

Run the collective from the driver (a `proc`), over the pool's column:

```text
total :: int;
N :: arche { v };
[8]N(8);

drive :: proc() {
  total := reduce(+, N.v);   // the whole-column fold, in the driver
}
```

## Why

The whole point of the `map`/collective split is that each has a different shape on parallel
hardware: a `map` is an embarrassingly-parallel element-wise pass (one GPU thread per element),
while a `reduce`/`scan` is a tree-fold and a `sort` is a global permutation — different dispatch
entirely. Mixing them would force the per-element kernel to also see the whole column, which
defeats the model. Keep kernels element-wise; run collectives over the column from the driver.
