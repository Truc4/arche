# W0023 — out-arg shadows out-param

A call **out-argument** written in the **colon** form `(name:)` is reported when
`name` is also the **out-parameter** of the enclosing proc. The colon *declares a
fresh local* called `name`; the call's result fills that local, and the proc's
out-param of the same name is **left unwritten** — a silent lost writeback (the
caller reads the zero-initialized out-param).

## The two out-arg forms

A proc-call out-list binds the callee's results:

```arche
foo(x)(r:)   // COLON  → declares a NEW local `r` to receive the result
foo(x)(r)    // no colon → writes an EXISTING place `r`
```

## What trips it

A wrapper that forwards a call's result into its **own** out-param, but uses the
colon (declaring) form instead of the plain (writing) form:

```arche
outer :: proc(x: int)(r: int) {
  inner(x)(r:);   // W0023 — `r:` shadows out-param `r`; `outer`'s `r` stays unset
}
```

At the call site `outer(10)(v:)` then reads `v` as `0`, because `outer` never
wrote its out-param.

## How to fix

Drop the colon so the call writes the out-param directly:

```arche
outer :: proc(x: int)(r: int) {
  inner(x)(r);    // writes out-param `r`  ✓
}
```

This is the form the stdlib wrappers use (`net.listen` → `net_listen(port)(s)`,
not `(s:)`).

## Not flagged

- `(name)` (no colon) — writes the existing out-param, the intended form.
- A colon name that is a **genuinely new** local, not an out-param:
  ```arche
  inner(x)(q:);   // `q` is a fresh local, not the out-param → fine
  r = q;
  ```
- `_` discards.

## Tuning

W0023 is a default-on lint. Silence a specific site with
`@allow(outarg_shadows_outparam)` on the enclosing decl if the shadowing local is
genuinely intended.
