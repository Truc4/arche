# E0046 — a `map` body is a sequence of column transforms

A `map` is a **per-element, branch-free, loopless kernel**: the runtime owns the iteration over the
matching pool, and each statement is a column transform (`col = expr`). That restriction is what lets the
same kernel source vectorize on the CPU today and run on the GPU later — so a `map` body admits **only**
column assignments. Control flow (`if`, `for`, `match`), loops, local bindings, calls, and `run` are not
transforms, so they are rejected here.

This isn't a hand-maintained ban list — it falls out of what a kernel *is*: anything that isn't a column
transform simply isn't part of the form.

## Instead

- **Conditionals → `select`** (branch-free, vectorizes, no GPU warp divergence):
  ```text
  // not:  if (pos.x > MAX) { vel.x = -vel.x; }
  vel.x = select(pos.x > MAX, 0.0 - vel.x, vel.x);
  ```
- **Control flow, loops, and effects (I/O, drawing) go in a `func`/`proc`**, called from the driver — not
  inside the kernel. The driver runs the `map` with `run`, then does its imperative work around it.
- **A whole-array operation** (sum, sort, draw) is a *collective*, not a kernel — use `reduce`/`scan`/`sort`
  or a device call on the columns, not a loop inside a `map`.

## Why

A loop or branch inside a kernel defeats the entire model: it forces the programmer to hand-write the map
(removing the runtime's freedom to vectorize / thread / GPU-dispatch), and a per-element branch diverges a
SIMD lane group / GPU warp. Keeping the body branch-free and loopless keeps it portable across backends.
